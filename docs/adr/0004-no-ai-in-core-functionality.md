# ADR 0004: No AI involvement in core functionality

**Status:** Accepted

## Context

DOT's end goal involves AI — a chat interface that answers site-specific questions. The question was how deeply AI should be embedded in the core pipeline: should the scanner use AI to interpret field types? Should gap detection use AI to suggest annotations? Should annotation drafts be AI-generated by default?

Embedding AI in core creates hard dependencies on external APIs, introduces latency and cost into admin operations, makes the tool non-functional without an API key, and creates data sovereignty concerns for agencies with privacy-sensitive clients.

## Decision

AI is external and minimal. The only AI touchpoints are:

1. **Query → response** — the chat interface sends one assembled payload, gets one response
2. **AI draft generation** — human-initiated, human-gated (parked for v1, see `dot_ai_draft`)

Scanning, gap detection, annotation storage, coverage reporting, and change detection are entirely AI-free. The site assembles context locally; AI only processes the final assembled payload on demand.

## Consequences

- DOT is fully functional without any API key — scanning, annotation, and reporting work standalone
- `dot_ai_context` (payload assembly) and `dot_ai_draft` (draft generation) are optional submodules; core value does not depend on them
- No RAG infrastructure needed for v1 — annotation sets for a typical Drupal site fit in a single prompt
- Agencies with strict data sovereignty requirements can use DOT without any data leaving the site
- The AI instruction layer must explicitly tell the model to say "I don't have specific information about that on this site" rather than guessing, to prevent confabulation
